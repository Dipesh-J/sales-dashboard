# Sales & Store Analytics — Full Stack Requirements

## Tech Stack

- **Backend:** Python, FastAPI, SQLAlchemy, PostgreSQL (or SQLite for dev)
- **Frontend:** React (Vite), Recharts/Chart.js, Material UI / Ant Design
- **Monorepo:** `/backend` and `/frontend` directories

---

## BACKEND (FastAPI)

### Data Model

```
Product: id, name, brand, category
Region: id, name
Store: id, region_id (FK → Region)
Sales: id, product_id (FK), store_id (FK), date, quantity, value
```

### Data Ingestion

- **Seed loader:** Script or CLI command to load initial sample data from CSV/Excel on first run.
- **Upload API:** Endpoint to upload Excel/CSV files and ingest into DB at runtime.
- Provide a sample dataset (seed data) covering at least 2 years, multiple brands, categories, regions.

#### Upload Endpoint

| Endpoint | Description |
|---|---|
| `POST /api/data/upload` | Accepts `.xlsx`, `.xls`, or `.csv` file upload. Parses, validates, and upserts into Products, Regions, Stores, Sales tables. |

**Upload handling requirements:**
- Accept file via `multipart/form-data`.
- Use `openpyxl` (xlsx) or `pandas` for parsing.
- Validate columns/schema before inserting — return clear error messages for malformed files (missing columns, bad data types, etc.).
- Handle duplicates gracefully (upsert or skip with a summary).
- Return a response summary: rows processed, rows inserted, rows skipped/errored.
- Support large files without blocking (consider background task with status polling for big uploads, optional bonus).

### API Endpoints

All endpoints return JSON. Use query params for filters.

#### Sales Metrics

| Endpoint | Description |
|---|---|
| `GET /api/sales/total` | Total sales by brand/product for a given month. Params: `month`, `year`, `brand?`, `product?` |
| `GET /api/sales/yoy` | Sales comparison: current year vs previous year. Params: `year`, `brand?`, `category?`, `region?` |
| `GET /api/sales/by-region` | Sales grouped by region. Params: `start_date`, `end_date`, `category?`, `brand?` |
| `GET /api/sales/by-category` | Sales grouped by category. Params: `start_date`, `end_date`, `region?` |
| `GET /api/sales/top-products` | Top N products by sales value. Params: `n`, `start_date`, `end_date` |
| `GET /api/sales/trend` | Sales trend over time (monthly aggregation). Params: `start_date`, `end_date`, `brand?`, `category?`, `region?`, `granularity` (month/quarter/year) |

#### Active Stores Metrics

**Definition:** A store is "active" if it has ≥1 sale (invoice) in the selected period.

| Endpoint | Description |
|---|---|
| `GET /api/stores/active` | Active store count by brand/product/month. Params: `month`, `year`, `brand?`, `product?` |
| `GET /api/stores/active/yoy` | Year-over-year active store comparison. Params: `year`, `brand?`, `region?` |
| `GET /api/stores/active/by-region` | Active stores grouped by region. Params: `start_date`, `end_date`, `brand?` |
| `GET /api/stores/active/trend` | Active stores trend by month. Params: `start_date`, `end_date`, `brand?`, `region?` |

#### Generic / Utility

| Endpoint | Description |
|---|---|
| `GET /api/filters/brands` | List all brands |
| `GET /api/filters/categories` | List all categories |
| `GET /api/filters/regions` | List all regions |
| `GET /api/filters/date-range` | Min and max dates in dataset |

### Business Logic

- Support aggregation at **month**, **quarter**, and **year** levels via a `granularity` query param.
- All list/tabular endpoints must support **pagination** (`page`, `page_size`).
- All date-filterable endpoints accept `start_date` and `end_date` (ISO format).

### Bonus (Optional)

- Swagger/OpenAPI docs (auto-generated by FastAPI — ensure it's enabled at `/docs`).
- In-memory or Redis caching for heavy aggregations.
- Role-based access (JWT auth with admin/viewer roles).

### Backend Deliverables

- Clean project structure with routers, services, models, schemas separated.
- `requirements.txt` or `pyproject.toml`.
- Seed data script/CSV files.
- README: architecture decisions, how to run, how to seed data.

## FRONTEND (React + Vite)

### Data Upload Page (Optional but Recommended)

- Simple page/modal with a file picker to upload `.xlsx` or `.csv`.
- Show upload progress and result summary (rows inserted, errors).
- Validate file type on the frontend before sending.
- After successful upload, prompt user to refresh dashboards or auto-refresh.

### Dashboard 1: Sales Overview

#### KPI Cards (top row)

- Total Sales (value, for selected filters/period)
- YoY Growth % (vs same period last year)
- Top Brand (by sales value in selected period)

#### Charts

- **Sales trend over time** — line or area chart (monthly). X-axis: time, Y-axis: sales value.
- **Sales by brand** — bar or stacked bar chart.
- **Sales by region** — horizontal bar chart (map is optional bonus).

### Dashboard 2: Active Stores Overview

#### KPI Cards

- Total Active Stores (for selected period)
- YoY Change (absolute or %)

#### Charts

- **Active stores trend by month** — line chart.
- **Active stores by region** — bar chart.

### Global Filters (Critical)

Filters must appear on every dashboard and dynamically update **all** visuals on change.

- **Date range:** month / quarter / year selector
- **Brand:** dropdown (multi-select or single)
- **Product category:** dropdown
- **Region:** dropdown

Filters should be populated from backend `/api/filters/*` endpoints.

### Drill-Down & Interactions

- **Click drill-down:** Clicking a brand bar drills into products under that brand. Clicking a region drills into stores/products in that region.
- **Hover tooltips:** Show value, %, and context on hover.
- **Sorting:** Allow sorting chart data (e.g., top contributors highlighted).

### Tech Requirements

- Component-based architecture (reusable chart wrappers, filter components, KPI cards).
- API integration via custom hooks or a service layer (axios/fetch).
- Proper state management (React Context or Zustand — no prop drilling for filters).
- Loading states and error handling for all API calls.

### Bonus (Optional)

- Export charts/data to CSV.
- Dark/light mode toggle.
- Responsive/mobile-friendly layout.
- Lazy loading for dashboard sections.

### Frontend Deliverables

- Clean component structure: `/components`, `/hooks`, `/services`, `/pages`.
- README: how to run, dashboard design decisions, libraries used.

## Evaluation Priorities

**Backend:** API correctness → clean architecture → data model → aggregation logic → performance.

**Frontend:** Dashboard clarity & analytical thinking → data visualization quality → filter/interaction UX → code organization → performance.

**Overall:** Analytics mindset (not CRUD). Clean, scalable, production-quality code.
